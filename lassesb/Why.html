Why "jcxz100"?

Hvorfor "jcxz100"?

Hvad betyder "jcxz100"?

JCXZ 100 er en Intel 8086/8088 CPU assembler instruktion, som betyder "Jump if register CX is Zero to absolute address CS:0100hex".
CX bruges ofte som tæller (C=counter, X=16 bit register).
Adresse CS:0100 er den adresse, som den første programinstruktion placeres på, når DOS loader en .COM fil.
Instruktionen JCXZ er et eksempel på CISC tankegangen (det modsatte af RISC), når den er værst; en-eller-anden ingeniør hos Intel har implementeret den, fordi han troede den ville være smart. I praksis bruges den dog ret sjældent - og når den gør, så er det næsten altid sammen med en unconditional JMP, hvilket indikerer, at der havde været mere brug for en JCXNZ instruktion...

Kombinationen "JCXZ 100" er meget usandsynlig at finde i virkelig software, både fordi JCXZ ikke bruges ret meget, og fordi man aldrig looper tilbage til adresse 100h.

Koden i billedet nedenfor vil løbe amok, når den gemmes som en .COM fil og dernæst køres fra kommandolinien. Det skyldes at CX så vil indeholde antallet af program-bytes, som blev loadet ind i RAM (altså 2). Da dét er forskeligt fra 0 vil "Jump" delen ikke blive udført.
CPU'en vil derfor loade de efterfølgende ukendte bytes og forsøge at udføre dem, hvilket med overvejende sandsynlighed går galt.
(Hvis man derimod indtaster instruktionen "JCXZ 100" i DEBUG og bare kører uden at gemme først, vil programmet loope uendeligt... det er jo så også skidt ;-) DEBUG starter nemlig med at sætte alle registre til 0.)

What does "jcxz100" mean?

JCXZ 100 is an Intel 8086/8088 CPU assembler instruction, short for "Jump if register CX is Zero to absolute address CS:0100hex".
CX is often used as a counter (C=counter, X=16 bit register).
Address CS:0100 is the address where the first instruction-code is placed when DOS loads a .COM file.
The JCXZ instruction is an example of CISC way of thinking (the opposite of RISC) at its worst; some engineer at Intel implemented it, because he thought it'd be nifty. In praxis it is rarely used - and when it is, it is almost always paired with an unconditional JMP, which indicates, that a JCXNZ instruction would have been more handy...

The combination "JCXZ 100" is very unlikely to be found in actual software, both because JCXZ isn't used much, and also because you never loop back to address 100h.

The code in the picture below will run amock when it is saved to a .COM file and then executed from the command line. The reason is that CX gets initiated to the number of program bytes that was loaded into RAM (i.e. 2). As this isn't 0 the "Jump" part won't be effectuated.
So therefore the CPU will load the succeeding unknown bytes and try to execute them, which has every probability of going wrong.
(If on the other hand you enter the instruction "JCXZ 100" in DEBUG and just execute it without saving it, it will loop infinitely... which of course is bad too ;-) This is because DEBUG initially sets all registers to zero.)
